## 二、解密原理

### 2.1 加密算法

微信 4.x 使用 **SQLCipher 4**：
- 算法：AES-256-CBC + HMAC-SHA512
- KDF：PBKDF2-HMAC-SHA512，**256,000 次迭代**
- 页大小：4096 字节
- reserved：80 字节（IV 16 + HMAC 64）
- 每个 .db 文件有独立的 salt（前 16 字节）和 enc_key

### 2.2 密钥获取

| 微信版本 | 方法 | 实现 |
|---|---|---|
| 3.x | 内存扫描 ASCII `x'hex'` 模式 | 稳定 |
| 4.0.x | 内存扫描（raw key 格式 `x'<64hex_key><32hex_salt>'`）| 本仓库 `decryptor.py` 实现 |
| 4.1.x | **内存扫描已失效**（密钥调用后立即 GC）| 需 wx_key DLL 注入 Hook |

**raw key 格式**：96 字符 hex = 64 字符 enc_key + 32 字符 salt

**校验**：对 page 1 做 HMAC 校验确认密钥正确性

### 2.3 平台差异

| 平台 | 内存读取 API |
|---|---|
| Windows | `ReadProcessMemory`（pymem） |
| Mac | Mach VM API（C 重写） |
| Linux | `/proc/<pid>/mem`（需 root） |

## 三、数据库结构（微信 4.x）

### 3.1 db_storage 目录

```
db_storage/
├── contact/contact.db          # 联系人
├── session/session.db          # 最近会话列表
├── message/message_0~13.db     # 消息（14个库，按会话分表）
├── message/message_resource.db # 消息附件资源
├── sns/sns.db                  # 朋友圈
├── favorite/favorite.db        # 收藏
├── emoticon/emoticon.db        # 表情
├── head_image/head_image.db    # 头像
└── ...
```

### 3.2 消息表 Msg_\<hash\>

| 字段 | 说明 |
|---|---|
| local_id | 本地自增主键 |
| **msg_svr_id** | 服务器全局唯一 ID（**增量游标**） |
| type | 1=文本 / 3=图片 / 34=语音 / 43=视频 / 49=复合 |
| is_sender | 0=收到 / 1=发出 |
| create_time | Unix 秒级时间戳 |
| message_content | ZSTD 压缩的消息内容 |
| message_blob | 附件数据（语音 SILK / 图片） |

### 3.3 路径

| 平台 | 路径 |
|---|---|
| Windows | `C:\Users\<用户名>\Documents\xwechat_files\<wxid>_<hash>\db_storage\` |
| Mac | `~/Library/Containers/com.tencent.xinWeChat/Data/...` |
| Linux | `~/Documents/xwechat_files/<wxid>/db_storage/` |

## 四、准实时增量解析

### 4.1 监控方式对比

| 方案 | 延迟 | CPU | 推荐度 |
|---|---|---|---|
| **WAL mtime 轮询** | ~100ms | 低 | ⭐⭐⭐⭐⭐ |
| 文件哈希轮询 | ~500ms | 中 | ⭐⭐⭐ |
| inotify 文件监听 | ~50ms | 极低 | ⭐⭐⭐⭐ |
| DLL 注入 Hook | <50ms | 极低 | ⭐⭐（封号风险） |

**关键约束**：微信 WAL 文件预分配 4MB 固定大小，**不能用 size 判断变化**，必须用 mtime。

### 4.2 增量查询 SQL

```sql
-- 记录上次解析的 msg_svr_id，每次只查新增
SELECT local_id, msg_svr_id, type, is_sender, create_time,
       message_content, message_blob
FROM Msg_e667c3a2...
WHERE msg_svr_id > :last_svr_id
ORDER BY create_time ASC, local_id ASC
LIMIT 1000;
```

### 4.3 准实时实现（本仓库 `message_extractor.py`）

1. 后台线程按 100ms 间隔轮询所有 `message_*.db-wal` 的 mtime
2. 检测到变化 → debounce 200ms（等写入完成）
3. 增量解密 → 按 msg_svr_id 游标查询新消息
4. 游标持久化到 SQLite（`parse_cursor` 表），重启可断点续传
5. 触发回调 `on_new_messages(talker, messages)`

## 五、语音 SILK 解码

### 5.1 微信 SILK 格式特点

标准 SILK：`#!SILK_V3` 开头，`\xFF\xFF` 结尾
微信 SILK：开头多了 1 字节 `\x02`，去掉了结尾 `\xFF\xFF`

### 5.2 解码流程（本仓库 `silk_decoder.py`）

```
微信 SILK → 去掉 0x02 头 + 加标准头/尾 → pilk 解码为 PCM → wave 转 WAV
```

- `pilk`：纯 pip 安装，自带 x64-win 预编译库（[foyoux/pilk](https://github.com/foyoux/pilk)）
- 采样率 24000Hz（微信语音标准）
- PCM → WAV 用 Python 标准库 `wave`，无需 ffmpeg

## 六、图片解密

微信图片是 `.dat` 加密文件，3 种格式：

| 格式 | 解密方式 |
|---|---|
| 旧版 XOR | 单字节异或，自动检测密钥 |
| V1 | AES-128-ECB + XOR，固定密钥 `cfcd208495d565ef` |
| V2 | AES-128-ECB + XOR，密钥从内存提取（需先点开图片） |

## 七、风险提示

### 7.1 法律风险
- PyWxDump 作者已因腾讯律师函删库，wechat-decrypt 仓库随时可能下架
- **建议内部 fork 备份，不公开传播，不商业化对外销售**
- 仅可解密本人或已获书面授权的员工的微信数据

### 7.2 封号风险
- wechat-dump-rs README 明确警告"可能存在封号风险"
- DLL 注入方案（4.1.x）封号风险更高，建议小号测试
- 内存扫描方案（4.0.x）风险较低，但非零

### 7.3 版本兼容
- 微信每次小版本更新都可能需要重新适配内存偏移
- 4.0.x 内存扫描在 4.1.x 已失效，需切换 wx_key DLL 注入
- 建议锁定微信版本，关闭自动更新
