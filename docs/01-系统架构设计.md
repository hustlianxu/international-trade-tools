# 01 - 系统架构设计

## 一、设计目标

为外贸业务员构建一个 AI 辅助工具，核心解决：
1. **微信沟通信息过载**：客户多、消息杂，重要需求易遗漏
2. **西语语音理解难**：客户主要用西语（含口音）发语音，业务员听不懂或听错
3. **跟进事项无系统**：报价、样品、订单、物流等待办散落在聊天记录中，缺乏统一推进

## 二、核心流程

```
PC 微信聊天 → 准实时解析(文字/语音/图片) → 语音转文字(西语+普通话)
    → DeepSeek 分析(需求提取/已办待办) → TODO 汇总 → 按天/小时提醒
```

## 三、分层架构

### 3.1 微信解析层（基于 ylytdeng/wechat-decrypt 二开）

| 组件 | 职责 | 技术 |
|---|---|---|
| `decryptor.py` | SQLCipher 4 数据库解密 | AES-256-CBC + HMAC-SHA512，内存扫描密钥 |
| `message_extractor.py` | 增量消息提取 + 准实时监听 | WAL mtime 轮询（100ms）+ msg_svr_id 游标 |
| `silk_decoder.py` | SILK v3 语音 → WAV | pilk（C 绑定）+ wave 标准库 |

**准实时机制**：
- WAL 文件预分配 4MB 固定大小，**不能用 size 判断变化**，必须用 mtime
- 轮询间隔 100ms，检测到变化后 debounce 200ms（等写入完成）
- 端到端延迟约 100ms（与 wechat-decrypt 原项目一致）

### 3.2 ASR 语音识别层

| 引擎 | 平台 | 成本 | 西语 WER | 适用 |
|---|---|---|---|---|
| MLX Whisper medium | Mac M3 | 0 元 | 4-6% | **首选**，隐私+免费 |
| 火山豆包 ASR 2.0 | 全平台 | ~7 元/月 | 5-7% | Windows 低配机 |
| gpt-4o-mini-transcribe | 全平台 | ~11 元/月 | 2.7-3% | 最高准确率 |

**关键设计**：
- 抽象接口 `ASREngine`，引擎可配置切换（config.yaml `asr.engine`）
- 语言留空自动检测（Whisper 内置 LID），适配中西混用场景
- `initial_prompt` 注入外贸热词（cotización/pedido/envío/muestra/pago），提升术语识别率

### 3.3 LLM 需求分析层

| 配置 | 值 |
|---|---|
| 模型 | DeepSeek V4-Flash（deepseek-chat） |
| 月费 | < 1 元（500条×300tokens） |
| 温度 | 0.3（业务理解需稳定） |
| 输出 | JSON（needs/done_items/todo_items/customer_mood） |

**需求分类体系**：inquiry(询价) / quotation(报价) / sample(样品) / order(订单) / logistics(物流) / payment(付款) / complaint(投诉) / other

### 3.4 TODO 提醒层

- 按 urgency（high/normal/low）+ deadline 排序
- 超期待办（超过 `overdue_days` 天）标红提醒
- 按客户分组汇总，支持按天/按小时提醒
- 提醒渠道：console（终端）→ 后续可扩展 wechat_file（写文件）/ 邮件 / 企微

### 3.5 存储层

本地 SQLite，4 张表：
- `parse_cursor`：增量解析游标（断点续传）
- `transcription`：语音转写缓存（避免重复转写）
- `analysis`：DeepSeek 分析结果
- `todo`：待办事项

## 四、数据流详解

### 4.1 一条语音消息的完整处理链路

```
1. 微信收到语音消息 → 写入 message_0.db 的 Msg_<hash> 表（type=34）
2. WAL 文件 mtime 变化 → monitor 100ms 内检测到
3. debounce 200ms → 增量解密 message_0.db
4. SQL 查询 msg_svr_id > 游标 → 提取新消息（含 blob_data = SILK 语音）
5. silk_decoder: SILK → 去掉 0x02 头 → pilk 解码 PCM → wave 转 WAV
6. ASR 引擎转写（MLX Whisper ~7秒/分钟音频）
7. 转写文本存入 transcription 表（缓存）
8. 拼入对话上下文 → DeepSeek 分析（JSON 输出需求+待办）
9. 待办写入 todo 表 → 等待提醒触发
```

### 4.2 一条文字消息的处理

```
1. 微信收到文字消息 → message_0.db（type=1，content 为 ZSTD 压缩）
2. WAL 变化 → 增量提取 → ZSTD 解压 content
3. 直接拼入对话上下文 → DeepSeek 分析
```

## 五、跨平台适配

| 平台 | 微信路径 | 密钥获取 | ASR 引擎 |
|---|---|---|---|
| Windows | `C:\Users\xxx\Documents\xwechat_files\` | pymem 内存扫描（4.0.x）/ DLL注入（4.1.x） | 火山豆包（云端） |
| Mac M3 | `~/Library/Containers/com.tencent.xinWeChat/` | Mach VM API | MLX Whisper（本地） |
| Linux | `~/Documents/xwechat_files/` | /proc/pid/mem | 火山豆包 |

## 六、扩展性设计

- **ASR 引擎可插拔**：新增引擎只需继承 `ASREngine` 并在 config 注册
- **LLM 可切换**：DeepSeek 可替换为 Qwen/GLM（同 OpenAI 兼容接口）
- **提醒渠道可扩展**：`reminder.channels` 配置数组，新增渠道实现 `send()` 方法
- **消息过滤可配置**：`watch_talkers` / `ignore_talkers` 白名单黑名单

## 七、成本核算（月费 < 100 元，实测 1-12 元）

| 项 | 方案 A（本地） | 方案 B（云端） |
|---|---|---|
| ASR | MLX Whisper = 0 元 | 火山豆包 500分钟 = 6.7 元 |
| LLM | DeepSeek = 0.2 元 | DeepSeek = 0.2 元 |
| 存储 | SQLite = 0 元 | SQLite = 0 元 |
| **合计** | **≈ 0.2 元** | **≈ 7 元** |

远低于 100 元/月预算。
